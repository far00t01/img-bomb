**IMG-Bomb💣** is a security testing tool designed for web application auditors and developers. It allows you to simulate Denial of Service (DoS) vectors based on image file processing, helping to identify resource exhaustion vulnerabilities in both disk space and memory/CPU allocation.

## The Concept Behind the Tool
Multimedia file processing is one of the most resource-intensive tasks for a web server. IMG-Bomb exploits the decompression logic of popular libraries (such as GD, Imagick, or ImageMagick) to evaluate whether a web application implements strict validation controls on user-uploaded files.

## Why Use IMG-Bomb?
The script allows you to generate two specific types of Proof of Concept (PoC) files to verify infrastructure resilience:

**Disk Space Exhaustion**
- Generates files with a legitimate PNG header but structured internally with null binary blocks (`\x00`).
- **Objective:** Verify if the web application's filters only validate the file extension or the magic bytes header without applying strict *Max Upload Size* limits, which could allow an attacker to fill up the server's storage with unnecessarily heavy files (ranging from megabytes to gigabytes).

**PNG Pixel Flood (Decompression Bomb)**
- Creates images with critical dimensions (by default 30,000 x 30,000 pixels) structured using high-frequency microscopic geometric patterns.
- **The Magic of Compression:** Thanks to the mathematical predictability of the pattern, the **Deflate** algorithm compresses the file to an extremely small footprint on disk (a few megabytes), easily bypassing shallow upload size filters.
- **Impact on the Server:** When the backend attempts to process, resize, or validate the image pixels using system libraries, the rendering engine is forced to deserialize the entire bitmap. This triggers a sudden allocation of **over 2.7 GB of RAM** and critical CPU spikes, exposing a lack of dimension limits (*Width/Height restrictions*) in the application.

## Main Features
- Minimizes the storage impact on the auditor's local machine while maximizing the decompression impact on the target.
- Clean visual layout with structured continuous tree-guides for the terminal.
- Allows instantaneous customization of pixel dimensions or binary sizes in KB, MB, or GB.
- Built on Python's standard library and powered exclusively by `Pillow`.

## Recommended Mitigation (For Developers)
If your web application is affected by the files generated by this script, implementing the following countermeasures is highly recommended:

### For Disk Space Exhaustion (Storage Protection)
1. Never rely solely on frontend restrictions. Ensure your backend validates the actual byte size of the incoming file block before saving it to storage.
2. Block oversized payloads at the gateway level before they even hit your application logic.
   - **Nginx:** Configure the `client_max_body_size` directive (e.g., `client_max_body_size 10M;`).
   - **Apache:** Use the `LimitRequestBody` directive to specify the maximum number of bytes allowed.
3. Implement temporary upload directories with restricted user permissions and execute routine background jobs (cron tasks) to purge unlinked, abandoned, or incomplete file chunks.

### For PNG Pixel Flood (Memory & CPU Protection)
1. Use lightweight streaming functions that only parse the metadata from the PNG `IHDR` chunk. Extract the image's width and height to discard images with absurd resolutions (e.g., 30,000 x 30,000) *before* attempting to deserialize or load the entire bitmap into memory.
2. Limit the maximum heap memory allocated to backend image processing workers via configuration files:
   - **PHP:** Restrict execution limits using `memory_limit` and `max_execution_time`.
   - **ImageMagick:** Enforce strict resource thresholds inside ImageMagick's `policy.xml` (e.g., limiting `<policy domain="resource" name="memory" value="256MiB"/>` and `<policy domain="resource" name="area" value="128MP"/>`).
